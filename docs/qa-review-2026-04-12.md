# Arc QB Sync — QA Review

**Date:** 2026-04-12  
**Version reviewed:** 3.1.2  
**Scope:** Code quality + functional logic across all PHP source files, JS, and docs; shortcode architecture across legacy and CPT branches  
**Files read:** `arc-qb-sync.php`, `qb-api.php`, `sync-courses.php`, `sync-events.php`, `sync-instructors.php`, `cache-rest.php`, `cpt-courses.php`, `cpt-events.php`, `cpt-instructors.php`, `shortcodes-courses.php`, `shortcodes-events-cpt.php`, `shortcodes-instructors.php`, `course-catalog.js`, `CHANGELOG.md`, `docs/setup.md`, `docs/field-mapping-courses.md`

---

## Summary

The plugin's architecture and core logic are sound. The sync engine pattern (fetch → upsert → ghost removal), the gatekeeper logic, and the admin UI are all well-structured and internally consistent. The main issues fall into two tiers: a small set of blockers that would prevent deployment or cause incorrect behavior in production, and a larger set of quality and consistency issues that should be addressed before the next version milestone.

### 3. `arc_qb_request()` treats a legitimate empty QB response as an error

**File:** `qb-api.php`, line 52

```php
if ( 200 !== $status || empty( $data['data'] ) || ! is_array( $data['data'] ) ) {
    return new WP_Error( 'arc_qb_bad_response', ... );
}
```

If the QB Records API returns a valid 200 response with zero records (e.g., no courses are marked public, or QB returns `{"data": [], ...}`), this condition evaluates `empty([])` as `true` and returns a `WP_Error`. That error propagates up through `arc_qb_sync_all_courses()` as a sync failure — "Sync finished with errors. 0 synced, 1 errors" — when in fact there were simply no records to sync.

**Resolution:**
```php
if ( 200 !== $status || ! isset( $data['data'] ) || ! is_array( $data['data'] ) ) {
    return new WP_Error( 'arc_qb_bad_response', ... );
}
```

Then in each `arc_qb_fetch_all_*()` caller, return the array directly (even if empty). The sync loops already handle empty arrays gracefully.

## High — Incorrect Behavior in Specific Scenarios

### 4. `[course_request_url]` shortcode bypasses context resolver

**File:** `shortcodes-courses.php`, line 322

All other course shortcodes use `arc_qb_get_course_post_id()`, which handles both native CPT pages and the legacy `?course-id=NNNN` URL pattern. `course_request_url` skips this and calls `get_the_ID()` directly:

```php
function arc_qb_shortcode_course_request_url( $atts ) {
    $post_id = get_the_ID(); // ← bypasses legacy fallback
```

If this shortcode is used on a page that serves content via the legacy `?course-id=` query parameter (rather than a CPT permalink), it returns empty while all other `[course_*]` shortcodes on that same page render correctly.

**Resolution:** Replace `get_the_ID()` with `arc_qb_get_course_post_id()`:

```php
function arc_qb_shortcode_course_request_url( $atts ) {
    $post_id = arc_qb_get_course_post_id();
    if ( ! $post_id ) {
        return '';
    }
    $record_id = intval( get_post_meta( $post_id, '_arc_qb_record_id', true ) );
    ...
```

### 5. Event `post_name` is title-derived — URL breaks on title change

**File:** `sync-events.php`, line 164

```php
$post_slug = sanitize_title( $title );
```

Courses use QB field 92 ("Slug for Website") as a stable, QB-managed slug. Events have no equivalent — the slug is derived from the event title on every upsert. If a title is corrected or updated in QB, the next sync changes the `post_name`, which changes the event's URL. There's no 301 redirect in place for the old URL.

**Options:** (1) Add a QB slug field to the events table and sync it, like courses do. (2) Only set `post_name` on insert, not on updates (don't include `post_name` in `$post_data` when `$post_id > 0`). Option 2 is the lower-effort fix.

## Medium — Code Quality and Consistency

### 7. `arc_qb_get_course_field()` is used as a generic accessor for events and instructors

**Files:** `sync-events.php`, `sync-instructors.php`

The function is named for courses but is used as the field-accessor for event and instructor records as well (e.g., `arc_qb_get_course_field( $record, 3 )` in `arc_qb_upsert_event()`). The function logic is generic — it just accesses `$record[$key]['value']` — but the name creates confusion for anyone reading `sync-events.php` cold.

**Resolution:** Rename `arc_qb_get_course_field()` to `arc_qb_get_field()` in `qb-api.php`, and update all callers. This is a two-line change in `qb-api.php` and a `replace-all` across three sync files. Keep the old name as an alias if backward compatibility with external callers matters.

---

### 8. Ghost removal and sync counts are confusing when upsert returns `0`

**Files:** `sync-courses.php`, `sync-events.php`, `sync-instructors.php`

When `arc_qb_upsert_course()` returns `0` (non-public record, no existing WP post), the calling loop in `arc_qb_sync_all_courses()` counts it as `$synced++`. In normal operation this doesn't happen (the fetch already filters to public=true), but in defensive reasoning, it means a no-op is indistinguishable from a successful sync in the count. This inflates the "synced" number. Minor in practice but can confuse debugging.

**Resolution:** Add an explicit check — if the return value is `0` and it's not a `WP_Error`, treat it as `$no_op++` rather than `$synced++`, and optionally report it separately in the admin UI.

---

### 9. Double DB query in all three preview functions

**Files:** `sync-courses.php` `arc_qb_preview_course_sync()`, and equivalents for events and instructors

Each preview function makes two separate `get_posts()` calls: one to get all existing posts (publish + draft) and a second to get only published posts for the ghost count. The ghost count could be derived from the first query by filtering in PHP. This is a minor efficiency issue that also makes the code harder to follow.

---

### 10. No error logging to WP debug log

Failed QB API calls return a `WP_Error`, which gets captured in `$messages[]` and shown in the admin sync UI — but only if an admin manually triggers a sync and sees the result. Failures from the Zapier webhook endpoint (`cache-rest.php`), ghost removal errors, and `wp_insert_post()` failures are never written to `WP_DEBUG_LOG`.

**Resolution:** Add `error_log()` calls at failure points:

```php
if ( is_wp_error( $result ) ) {
    error_log( '[arc-qb-sync] Upsert failed for QB record ' . $qb_record_id . ': ' . $result->get_error_message() );
}
```

### 13. `arc_qb_format_duration()` returns raw value for falsy input instead of empty string

**File:** `qb-api.php`, line 109

```php
if ( ! is_numeric( $ms ) || intval( $ms ) <= 0 ) {
    return esc_html( (string) $ms );
}
```

If `$ms` is an empty string (QB didn't return a value), this returns `esc_html('')` — fine. ## Shortcode Architecture — Two-Branch Review

The plugin intentionally runs two parallel shortcode systems. This section maps the architecture, confirms where the branches are safely separated, and flags specific issues where they aren't.

### Architecture overview

**Legacy branch — live QB fetch on page load**

- Pages: `/training-details/?event-id=NNNN`
- Data source: `events.php` → `arc_td_get_current_record()` → QB API call on each page load
- Shortcode prefix: `event_*`
- File: `shortcodes-events.php`

**CPT branch — reads from WordPress post meta**

- Pages: native CPT permalinks (`/course/[slug]/`, `/training/[slug]/`, `/instructor/[slug]/`)
- Data source: WP post meta, populated by the sync engine
- Shortcode prefixes: `course_*`, `arc_event_*`, `instructor_*`
- Files: `shortcodes-courses.php`, `shortcodes-events-cpt.php`, `shortcodes-instructors.php`

### Courses are already unified — events are not

This is the most important architectural distinction to understand:

**Courses** do not have a separate legacy shortcode branch doing live QB fetches. `shortcodes-courses.php` serves _both_ legacy `?course-id=NNNN` pages and native CPT pages through a single context resolver (`arc_qb_get_course_post_id()`). That resolver tries the current CPT post first, then falls back to looking up the CPT post by `_arc_qb_record_id` meta when a `?course-id=` parameter is present. One shortcode set, one data source (WP meta). Clean.

**Events** are genuinely split. Legacy pages use `[event_*]` shortcodes that make live QB calls. CPT pages use `[arc_event_*]` shortcodes that read WP meta. These two sets are not interchangeable — `[arc_event_dates]` on a legacy page returns empty; `[event_dates]` on a CPT page makes an unnecessary API call to QB.

**Instructors** are CPT-only. There is no legacy live-QB branch for instructors.

### What's safe

There are no shortcode name collisions between the two branches. `event_*` and `arc_event_*` do not share any names, and all three CPT prefixes (`course_*`, `arc_event_*`, `instructor_*`) are distinct from all legacy names. Both branches can coexist in the same WordPress instance without conflict.

### Issues found

**Issue S1 — Flyer shortcode naming inconsistency across branches**

All URL-returning shortcodes in the CPT branch follow the `_url` suffix convention:
`[arc_event_reg_url]`, `[arc_event_image_url]`, `[arc_event_featured_image_url]`, `[arc_event_hero_image_url]`, `[arc_event_flyer_url]`.

The legacy equivalent is `[event_flyer]` — no `_url` suffix. This is the only URL-returning legacy shortcode that breaks the pattern. Not a runtime conflict, but it's a mental mapping trap: a developer moving from legacy to CPT would expect `[arc_event_flyer]` (by analogy from every other `event_X` → `arc_event_X` mapping) and get empty instead.

_No code fix needed — just document the exception explicitly._

---

**Issue S2 — Generic shortcode parameter APIs are fundamentally different between branches**

| | Legacy | CPT |
|---|---|---|
| Event shortcode | `[event_field id="45"]` | `[arc_event_field meta="_arc_event_dates"]` |
| Course shortcode | `[course_field id="45"]` | _(same shortcode, same file)_ |
| Instructor shortcode | _(none)_ | `[instructor_field meta="_arc_instructor_bio"]` |

The legacy `[event_field]` takes a numeric QB field ID. The CPT `[arc_event_field]` takes a WP meta key string. These look structurally similar but have entirely different parameter semantics. Using `[arc_event_field id="45"]` silently returns empty — no error, no indication of the mismatch.

The `[course_field]` generic shortcode is not split between branches (it's CPT-only), so there's no confusion there.

_This is worth adding to the docs (webhook-zapier.md or a new shortcode reference) to prevent template-building errors._

---

**Issue S3 — Boolean output differs between branches for `is_multiday` and `is_multisession`**

Legacy shortcodes (`shortcodes-events.php`, lines 236 and 247):
```php
return ! empty( $value ) ? '1' : '0';
```

CPT sync (`sync-events.php`, lines 215 and 218):
```php
( $is_multiday && 'false' !== strtolower( (string) $is_multiday ) ) ? '1' : '0'
```

The legacy version uses `! empty()`. In PHP, `! empty("false")` evaluates as `true`, so if QB returns the string `"false"` for a checkbox field, the legacy shortcode returns `'1'` (incorrect). The CPT sync correctly guards against the `"false"` string. The stored meta values and the live-fetch values could therefore disagree on the same QB record.

_Fix the legacy shortcodes to use the same guard:_
```php
$value = arc_td_get_field_value( 453 );
return ( $value && 'false' !== strtolower( (string) $value ) ) ? '1' : '0';
```

**Issue S4 — Legacy event pages have no access to Image Asset lookup URLs**

The legacy `arc_td_get_current_record()` in `events.php` fetches FIDs: 3, 14, 19, 29, 45, 89, 267, 271, 361, 413, 440, 449, 450, 453, 454, 458, 461. It does not include FID 464 (Featured Image lookup) or FID 466 (Hero Image lookup).

Legacy pages therefore only have access to `[event_image_url]` (FID 461, the legacy manual URL field). The Image Asset lookup URLs (`[arc_event_featured_image_url]`, `[arc_event_hero_image_url]`) are only available on CPT pages.

This isn't a bug — it's an intentional migration gap — but it means any image quality improvement from the Image Assets table won't appear on legacy event detail pages until those pages are cut over to CPT templates.

_No fix required. Document it as a known limitation of the legacy branch._

---

**Issue S5 — Structured instructor slots are CPT-only; no legacy equivalent exists**

The legacy `[event_instructors]` shortcode returns FID 271 as a plain text string (e.g. "Jane Smith, Bob Jones"). The CPT branch has structured per-slot data: `[arc_event_instructor1_name]`, `[arc_event_instructor1_headshot_url]`, `[arc_event_instructor1_headshot_alt]`, and equivalents for slots 2 and 3.

Legacy event pages cannot access per-instructor headshots or alt text. This is expected — the structured fields were added as part of the CPT migration — but it means the template designs for legacy pages and new CPT pages will necessarily differ in their instructor display capabilities.

_No fix required. Confirm that legacy event templates don't need headshots before closing this out._

---

**Issue S6 — `arc_qb_get_public_courses()` in `courses.php` has no active callers**

`courses.php` defines two functions: `arc_qb_get_course()` (single-record fetch) and `arc_qb_get_public_courses()` (full catalog fetch with 15-minute transient). Neither is called by any current shortcode or sync file — the course catalog is now served by the CPT Loop Grid, not by a shortcode that calls QB directly.

Both functions are preserved for backward compatibility (and `courses.php` is required by `arc-qb-sync.php`), but they appear to be dead code. `arc_qb_get_course()` is the more likely candidate to be called externally, since it was the original single-course detail page engine.

_Confirm nothing on the live site calls these functions directly (e.g., via a custom template or external snippet). If not, they can be removed in a future version._
