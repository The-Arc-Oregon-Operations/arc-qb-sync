## Goal

Debug the v3.7.0 Option A image pipeline. First live test: uploaded an image via QB file attachment field, ran sync, confirmed something reached WP but with two defects.

## What Was Done

### Bugs found and fixed (→ v3.7.1)

Both bugs were in `arc_qb_download_image_file()` in `qb-api.php`.

**Bug 1 — QB Files API returns base64, not raw binary.**
The response body from `GET /v1/files/` is base64-encoded text. The original code wrote it directly to a temp file, so WP received base64 ASCII rather than valid image binary. Result: blank preview in WP media library, 39 KB text file stored as an attachment.

Fix: `base64_decode( $body, true )` before `file_put_contents`. Strict mode — if decode returns `false` (invalid base64), falls through to raw body unchanged.

**Bug 2 — No file extension on the sideloaded attachment.**
The `name` key in the file array was built from `$ia_filename` (e.g. `my-course-featured` with no extension). WP uses `name` to set the destination filename and determine file type — without an extension, WP can't generate thumbnails or process the image correctly.

Fix: resolve final filename in this priority order:
1. `Content-Disposition` header from QB response — preserves the original uploaded filename and extension (e.g. `tips-supporters-hero.png`)
2. Extension derived from `Content-Type` header (`image/png` → `.png`, etc.) appended to the `$ia_filename` base
3. `$ia_filename` as-is (extensionless fallback — only reached if QB sends neither header nor a recognizable MIME type)
