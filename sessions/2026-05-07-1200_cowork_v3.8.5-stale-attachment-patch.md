## Goal

Troubleshoot why Course 390 (Tips & Tools for Supporters) had no featured image set in WP despite the image existing in WP Media and the QB Image Asset record appearing correctly wired.

## Investigation

Traced the pipeline in `arc_qb_sync_set_featured_image` (`qb-api.php`):

1. Fast path fires when FID 111 (Featured Image - Attachment ID lookup) returns a non-zero value.
2. QB Image Asset #17 had Attachment ID = 38524, Processing Status = Processed — writeback had occurred.
3. Confirmed WP attachment 38524 no longer exists (`/wp-admin/post.php?post=38524` → "item does not exist").
4. Root cause: the attachment was deleted from WP Media after the initial sideload. QB still held the old ID. Fast path called `set_post_thumbnail($post_id, 38524)` on every sync — `set_post_thumbnail` does not validate attachment existence, so it silently wrote a stale `_thumbnail_id`. WP admin showed "no featured image."

## Manual fix (applied by Alan)

- Located current WP attachment for the image: ID **38565**.
- Updated QB Image Asset #17 FID 24 (Attachment ID) from `38524` → `38565`.
- Confirmed featured image now displays on Course 390 post (WP ID 38492) after next sync.
- Other courses audited manually — no additional stale IDs found.

## Code change

**File:** `arc-qb-sync/includes/qb-api.php`
**Function:** `arc_qb_sync_set_featured_image()`
**Change:** Added `wp_attachment_is_image()` guard to fast path. If the stored attachment ID no longer exists in WP, the function logs the stale ID and falls through to the sideload path to re-acquire. If sideload succeeds, the correct new ID is written back to QB. If sideload also fails, the stale case is at least logged rather than silently writing broken metadata.
