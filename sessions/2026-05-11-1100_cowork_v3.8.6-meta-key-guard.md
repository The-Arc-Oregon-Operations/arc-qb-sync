## Goal

Restore rendering of legacy `/training-details/?event-id=NNNN` pages on thearcoregon.org. Alan reported that the page at `https://thearcoregon.org/training-details/?event-id=1827` was "missing most of the content" today, four days after the v3.8.0–v3.8.5 batch of patches shipped on 2026-05-07.

## Investigation

The 2026-05-07 session note (`2026-05-07-1200_cowork_v3.8.5-stale-attachment-patch.md`) captured only the v3.8.5 patch (stale-attachment guard in `arc_qb_sync_set_featured_image`). Eight commits actually landed across May 7–8: a pre-3.8.0 image-pipeline fix, v3.8.0 through v3.8.5, a `.gitignore` update, and a gitcleanup. The session note had documented only the last of these.

Code path traced for the legacy URL:

1. `/training-details/?event-id=1827` is a regular WP page (not the `arc_event` CPT).
2. Page content includes `[event_*]` shortcodes (Group A — registered in `shortcodes-events-cpt.php`, read from WP meta on the matching `arc_event` CPT post) and deprecated alias shortcodes like `[venue_name]`, `[instructors]`, `[training_cost]` (Group B — registered in `shortcodes-events.php`, hit QB directly via `wp_remote_post`).
3. Every Group A shortcode calls `arc_qb_get_event_post_id()` in `shortcodes-events-cpt.php`, which falls through to a `get_posts()` lookup by `_arc_qb_event_id` meta when the current page isn't an `arc_event` CPT.
4. That `get_posts()` fires `pre_get_posts`, which fires `arc_qb_default_event_order()` in `cpt-events.php`.
5. v3.8.4 ("fix CPT default sort — remove orderby guard that let Elementor override") had removed the early-return guard that previously kept this hook from running on queries with an explicit orderby. The current body unconditionally does:

   ```php
   $query->set( 'meta_key', '_arc_event_start_date' );
   $query->set( 'orderby',  'meta_value' );
   $query->set( 'order',    'ASC' );
   ```

6. The first `set` call clobbers the lookup's `meta_key` (originally `_arc_qb_event_id`), turning the query into "find any `arc_event` where `_arc_event_start_date` equals 1827" — zero results.
7. `arc_qb_get_event_post_id()` returns `false`, every Group A shortcode returns an empty string, and the page renders mostly blank. Group B shortcodes continue to work because they bypass WP_Query entirely.

The function's docblock still read *"Only fires when orderby has not already been set on the query"* — stale relative to the body, consistent with an incomplete refactor. v3.8.4's commit message confirmed the removal was deliberate and applied to course, instructor, AND arc_event hooks.

## Code change

**Files touched:**

- `arc-qb-sync/includes/cpt-events.php` — `arc_qb_default_event_order()`: added `if ( $query->get( 'meta_key' ) ) { return; }` guard. Docblock rewritten to describe actual behavior.
- `arc-qb-sync/includes/cpt-courses.php` — `arc_qb_default_course_order()`: same guard added defensively. The function does not currently clobber `meta_key` (it only sets `orderby`), so the bug does not manifest for `?course-id=` lookups today, but the guard keeps the three functions consistent and prevents a future regression. Docblock updated.
- `arc-qb-sync/includes/cpt-instructors.php` — `arc_qb_default_instructor_order()`: same defensive guard. Docblock updated.
- `arc-qb-sync/arc-qb-sync.php` — header `Version:` and `ARC_QB_SYNC_VERSION` constant bumped 3.8.5 → 3.8.6.
- `CHANGELOG.md` — 3.8.6 entry added.

**Why `meta_key` guard rather than restoring the `orderby` guard:** v3.8.4's intent was to ensure the CPT's default sort overrides Elementor's archive / Loop Grid default. Elementor archive queries do not set `meta_key`, so a `meta_key` guard is invisible to them — v3.8.4's behavior is preserved. The previous `orderby` guard would not have helped the legacy lookup anyway (the lookup doesn't set orderby).
